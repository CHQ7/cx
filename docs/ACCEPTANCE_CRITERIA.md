# cx 分阶段验收标准

本文档定义了 `cx` 项目从 MVP 到完整交付的五个阶段的验收标准。每个阶段完成后，必须通过所有验收项，方可进入下一阶段。

---

## 第一阶段：MVP 核心

**目标**：Agent 循环 + 文件读写工具 + 基础 TUI + OpenAI 适配。

### 验收项

| # | 验收项 | 验证方式 | 状态 |
|---|--------|----------|------|
| 1.1 | `cargo build --release` 无错误，生成二进制 `cx` | 运行命令 | ☐ |
| 1.2 | 执行 `./cx` 启动 TUI，显示输入框和空对话区域 | 手动运行 | ☐ |
| 1.3 | 输入“创建一个 test.txt，内容为 hello world”，文件确实创建且内容正确 | 检查文件系统 | ☐ |
| 1.4 | 输入“读取 test.txt”，TUI 显示文件内容 | 观察 TUI 对话区域 | ☐ |
| 1.5 | 输入“把 test.txt 中的 world 替换为 cx”，文件内容变为 “hello cx” | 检查文件内容 | ☐ |
| 1.6 | 配置 `~/.cx/config.toml`（api_key, model），重启后生效 | 修改配置后重启，观察模型调用 | ☐ |
| 1.7 | 空闲时内存占用 < 50 MB，启动到出现输入提示 < 100ms | 使用 `time` + `htop`/`ps` | ☐ |
| 1.8 | Agent 循环能正确处理无工具调用的普通对话（如“写一首诗”） | 输入对话，观察 LLM 回复 | ☐ |

> 备注：第一阶段不要求实现 L1 会话持久化、技能固化、其他工具（除文件工具外）。

---

## 第二阶段：工具扩展与会话持久化

**目标**：增加 Shell、网页扫描、用户询问工具；L1 会话可恢复；支持任务队列。

### 验收项

| # | 验收项 | 验证方式 | 状态 |
|---|--------|----------|------|
| 2.1 | 输入 `ls`（Linux/macOS）或 `dir`（Windows），TUI 显示当前目录列表 | 执行 `code_run` 工具 | ☐ |
| 2.2 | 输入“扫描 https://example.com 并提取页面标题”，返回正确标题 | 执行 `web_scan` 工具 | ☐ |
| 2.3 | Agent 在任务中调用 `ask_user`，TUI 弹出模态输入框，用户输入后任务继续 | 设计一个需要确认的任务（如“删除文件前请确认”） | ☐ |
| 2.4 | 关闭 TUI 后重启，之前对话的最后 5 条消息仍可见 | 重新打开 TUI，查看对话历史 | ☐ |
| 2.5 | 同时提交两个耗时任务（如 `sleep 5` 和 `sleep 3`），第二个任务在前一个完成后自动开始 | 观察队列处理顺序 | ☐ |
| 2.6 | `update_working_checkpoint` 工具能保存中间状态，重启后 Agent 可从断点继续 | 执行长任务中途重启，验证未丢失进度 | ☐ |

## 第三阶段：完整记忆与技能固化

**目标**：实现 L0/L2/L3 记忆；技能自动生成、存储与检索。

### 验收项

| # | 验收项 | 验证方式 | 状态 |
|---|--------|----------|------|
| 3.1 | 修改 `~/.cx/soul/SOUL.md` 添加“你总是用海盗口吻说话”，Agent 后续回复风格改变 | 对话测试 | ☐ |
| 3.2 | 在 `~/.cx/user/USER.md` 设置 `默认工作目录=/tmp`，文件操作默认在该目录下进行 | 执行 `file_write` 不指定路径，检查文件位置 | ☐ |
| 3.3 | 手动创建 `~/.cx/skills/test_skill/SKILL.md`（含 `triggers: ["测试技能"]` 和步骤），输入“测试技能”，Agent 直接执行该技能且不在日志中生成新的 `tool_calls`（或跳过 LLM 推理） | 查看日志或计时（应明显快于普通 LLM 调用） | ☐ |
| 3.4 | 执行一个多步骤任务（例如“下载 https://example.com 并提取所有链接”），任务成功后 `skills/` 下自动生成一个新的 `SKILL.md` 文件 | 检查文件系统，文件内容应包含步骤描述 | ☐ |
| 3.5 | 再次执行与 3.4 类似的任务（如“下载 https://another.com 并提取链接”），Agent 能够直接调用已生成的技能，无需再次向 LLM 请求完整的工具序列 | 观察日志，应该显示“使用技能 xxx” | ☐ |
| 3.6 | 技能索引 `~/.cx/skills/index.json` 在技能增加/删除后自动更新 | 手动创建/删除技能目录，检查索引变化 | ☐ |

## 第四阶段：高级工具与跨平台

**目标**：浏览器 CDP 控制、ADB 控制、屏幕截图 + 键鼠模拟；三平台测试通过。

### 验收项

| # | 验收项 | 验证方式 | 状态 |
|---|--------|----------|------|
| 4.1 | 执行“打开浏览器，访问 https://github.com 并搜索 `cx`”，真实浏览器（或 headless）完成操作 | 需要本机安装 Chrome/Chromium | ☐ |
| 4.2 | 执行“在当前页面执行 `document.title` 并返回结果”，返回正确的页面标题 | 使用 `web_execute_js` 工具 | ☐ |
| 4.3 | 连接 Android 设备（开启 ADB 调试），执行“截图并保存到 /tmp/screen.png” | 检查文件是否生成 | ☐ |
| 4.4 | 执行“获取屏幕截图并保存到 /tmp/capture.png” | 使用 `xcap` 截图，检查文件 | ☐ |
| 4.5 | 执行“模拟键盘输入 'Hello cx' 后按回车” | 在文本编辑器中观察输入效果（需谨慎测试） | ☐ |
| 4.6 | 在 **Windows 10/11**、**macOS 14+**、**Ubuntu 22.04** 上分别编译并运行所有工具（浏览器和 ADB 需预装依赖，但 `cx` 本身不 panic） | 三台物理机或虚拟机测试 | ☐ |

> 注意：4.5 可能干扰用户，可在专门的测试环境中进行，或设计为仅在不活跃窗口上模拟。

## 第五阶段：打磨与交付

**目标**：性能、健壮性、文档、测试覆盖率全面达标。

### 验收项

| # | 验收项 | 验证方式 | 状态 |
|---|--------|----------|------|
| 5.1 | 启动 TUI 到出现输入提示符 < 100ms（release 模式） | `time -p ./target/release/cx --help`（测量实际启动至 render） | ☐ |
| 5.2 | 执行一个无需工具的任务（如“写一首四行诗”），端到端延迟不超过 LLM API 延迟 + 200ms | 对比相同请求用 `curl` 直接调用 API 的时间差 | ☐ |
| 5.3 | 任意工具超时（如 `code_run` 执行 `sleep 60` 且超时设置为 5 秒）能捕获错误并恢复，Agent 循环不崩溃 | 触发超时，观察错误提示和后续正常对话 | ☐ |
| 5.4 | 按 Ctrl+C 时，进程立即退出，无残留子进程（如浏览器进程、adb 进程） | `ps aux | grep -E "(chrome|adb|cx)"` 检查 | ☐ |
| 5.5 | `cargo test` 全部通过，且 `cargo tarpaulin` 显示核心模块（`src/agent`、`src/memory`、`src/tools`）行覆盖率 ≥ 80% | 运行测试和覆盖率工具 | ☐ |
| 5.6 | 提供 `README.md`（包含安装、配置、运行示例），`docs/ARCHITECTURE.md`（模块设计），每个原子工具的使用说明 | 文档审阅 | ☐ |
| 5.7 | 三个端到端示例场景（整理下载文件夹、网页摘要、代码仓库总结）能够一次性执行成功 | 运行示例脚本或按文档手动操作 | ☐ |
